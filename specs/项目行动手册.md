# 项目行动手册 — Agent 执行总纲领

> 小智机械臂 VLA 集成项目 | 分支 `feature/robot-arm-vla`
>
> 本手册是 Claude Code Agent 执行所有开发任务的最高准则。

---

## 1. 五条核心原则

### 原则一：单点职责

每个功能点写在独立文件中，先读 specs/ 下的需求文档再编写代码。

| 技术栈 | 代码存放位置 |
|--------|-------------|
| ESP32 C++ 固件 | `main/boards/xiaozhi-custom/`（业务功能）、`main/protocols/`（协议）、`main/audio/`（音频）等 |
| Python 服务器端 | `c:\Users\24628\xiaozhi-esp32-server\main\xiaozhi-server\` 对应子目录 |
| 测试脚本 | `tests/*.py`（pytest） |
| 规格文档 | `specs/*.md` |

**禁止**：单文件堆砌全部功能、多功能混写。

### 原则二：原子化产出

每次只推进 1 个功能点（对应 plan 中的 F# 编号），完成后在 `progress.md` 中标记为「✅ 已完成」，再进入下一个。

### 原则三：检查点（Checkpoint）

写完一个功能点的代码后，必须立即编写对应的验证逻辑：

- **ESP32 固件**：通过串口监视器验证，或编译零错误零警告作为检查点
- **Python 服务器端**：编写 pytest 单元测试
- **配置文件**：重启服务后功能验证

以验证通过作为功能验收标准。

### 原则四：反馈循环（Feedback Loop）

编写完代码/测试后，立即执行验证：
1. 运行测试命令（`pytest tests/test_xxx.py` 或固件编译 `bash esp_idf_build.sh build`）
2. 若失败，读报错信息 → 分析原因 → 修复代码
3. 再次运行测试，直至通过

### 原则五：任务熔断

如果同一个功能点经过 **3 次**自我修复后测试仍不通过：
1. 停止循环
2. 在 `progress.md` 记录详细错误信息和已尝试的修复方案
3. 向用户求助，**禁止强行跳过**

## 2. 标准执行行动链

```
1. 读取 specs/ 目录文档，明确当前功能点需求
2. 在对应目录编写功能代码
3. 编写配套验证逻辑（测试/编译/手动检查）
4. 执行验证命令
5. 观察结果；失败则定位问题并修改
6. 重新执行验证，直至通过
7. 测试通过后，更新 progress.md；读取下一个功能点
```

## 3. 项目逻辑分层

由于 ESP-IDF 构建系统约束（CMakeLists.txt 通过 `file(GLOB)` 收集 `main/` 下源文件），C++ 源码必须保持在 `main/` 目录下。采用逻辑分层映射：

| 逻辑层 | 物理位置 | 说明 |
|--------|----------|------|
| **specs/** | `specs/*.md` | 功能规格说明文档 |
| **core/** | `main/protocols/`, `main/audio/`, `main/led/`, `main/display/` | 与业务无关的底层驱动和协议 |
| **features/** | `main/boards/xiaozhi-custom/`, `main/text_console/` | 机械臂控制、文字交互等业务功能 |
| **server/** | `c:\Users\24628\xiaozhi-esp32-server\main\xiaozhi-server\` | OpenClaw 服务器端代码 |
| **tests/** | `tests/*.py` | pytest 测试脚本 |

## 5. 代码规范

### C++（ESP32 固件）
- 遵循周边代码风格，不引入新的命名约定
- 新功能优先放在 `main/boards/xiaozhi-custom/` 下
- 通过 `main/CMakeLists.txt` 的 `file(GLOB)` 自动发现，无需手动注册

### Python（服务器端）
- 类型注解：所有函数参数和返回值
- 文档字符串：Google 风格
- 错误处理：显式 try/except，记录日志

### 配置文件
- 敏感信息（API Key 等）不写入 specs/
- 配置模板和示例写入 specs/，实际配置放在服务器 config.yaml
