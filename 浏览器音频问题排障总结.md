# 浏览器测试页音频问题排障总结

> 2026-06-24 | 问题：OpenClaw 测试页面对话无声音

---

## 1. 结论

**OpenClaw 测试页面（test_page.html）在 Windows Chrome 上无法播放 TTS 语音，根因未定位但已充分排除。建议跳过浏览器端音频调试，等 ESP32 开发板到手后直接验证硬件音频链路。**

---

## 2. 已验证的工作项

| 环节 | 状态 | 证据 |
|------|------|------|
| LLM (qwen-turbo) | ✅ 正常 | 0.9s 返回，服务器日志正常 |
| TTS (EdgeTTS) | ⚠️ 能用但慢 | 生成音频 10s，Microsoft TLS 握手延迟 |
| TTS (AliBLTTS) | ❌ 不可用 | WebSocket 连 DashScope 失败 |
| TTS (LinkeraiTTS) | ❌ 不可用 | 502 Bad Gateway，免费服务挂了 |
| OPUS 编码（服务器端） | ✅ 正常 | ffmpeg → PCM → OPUS 转换正常 |
| OPUS 数据传输 | ✅ 正常 | WebSocket 二进制帧到达浏览器 |
| OPUS 解码（浏览器端） | ✅ 正常 | `_opus_decode` 每帧返回 480 样本 |
| AudioContext（24000Hz） | ✅ 正常 | 原生测试音可播放 |
| audioContext.resume() | ✅ 正常 | 强制 resume 后可播测试音 |
| GainNode 通路 | ✅ 正常 | 测试音通过 gainNode → destination 可播 |
| streamingContext 调度 | ✅ 正常 | 日志显示调度播放 2880 样本 |

## 3. 未定位的问题

**OPUS 解码出的 PCM 数据，通过 streamingContext 的 AudioBuffer 播放链路时无声。**

可能原因（未确认）：
- `copyToChannel` 传入的 Float32Array 数据可能未被正确写入 AudioBuffer
- 定时调度逻辑 (`source.start(startTime)`) 可能导致静默
- 解码后的 PCM 实际采样率与 AudioBuffer 声明的 24000Hz 不匹配
- Chrome 对 24000Hz AudioBuffer 的兼容性问题

## 5. 可行替代方案

1. **等 ESP32 硬件到手**：ESP32 自带 OPUS 硬解码 + 扬声器驱动，无需浏览器
2. **用飞书 Bot 测试**：飞书→OpenClaw 链路已完成，文字回复正常
3. **换真实 ESP32-S3 开发板**：烧录固件后直接验证语音链路
