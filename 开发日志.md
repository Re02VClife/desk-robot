### 服务端关键修复（按排查难度排序）

| # | 问题 | 症状 | 根因 | 修复 |
|---|------|------|------|------|
| 1 | **async 缺 await** | `move_arm` 从不执行，无日志 | `plugin_executor.py` 调协程没 await | 加入 `await` 检查 |
| 2 | **has_tool 不匹配** | "工具不存在" | 工具名 `robot.arm.gripper` 被净化为 `robot_arm_gripper` | 同时接受两种格式 |
| 3 | **OTA URL 指向云端** | ESP32 连 `mqtt.xiaozhi.me` | sdkconfig OTA URL 未修改 | 改到本地 `10.183.125.233` |
| 4 | **IP 跨子网** | ESP32 无法访问服务器 | PC 和 ESP32 不在同一网段 | 改到同一 WiFi 热点 |
| 5 | **Python 缓存** | 代码改了不生效 | `__pycache__` 旧版本 | 清除所有 `.pyc` |
| 6 | **多步指令丢失** | "归位然后抬起"只执行归位 | 正则漏了连接词 `再` | 添加 `再` 到连接词 |
| 7 | **中文数字不识** | "两秒后" 不匹配 | 正则只认 `\d+` | 添加 `[一二三四五...]` 映射 |
| 8 | **第二条指令超时** | 15s 超时对 LLM 太短 | `text_timeout_tick_ = 15` | 改到 60s |
| 9 | **重连时文字丢弃** | `Cannot send text in state 4` | connecting 状态无处理 | 排队+自动补发 |

### move_arm 重构
- **之前**：硬编码 90° 基准，未指定的关节跳变 → 扫桌子危险
- **之后**：先调 `get_status` 获取当前位置 → 只改用户指定的关节 → 其余保持不动
- 支持多步指令："X 然后 Y 两秒后 Z" → 自动拆分顺序执行
- 支持中文数字："两秒"/"三秒"
- 静音模式：`Action.RESPONSE` + silent（不调 LLM 不生成语音）

### ARM_KEYWORDS 移除
- **之前**：检测到机械臂关键词 → 直接路由到 `move_arm`，LLM 无法插手
- **之后**：所有指令交 LLM 推理，LLM 自己决定调 `move_arm` 还是直调 MCP 工具
- 复杂指令（"抓取动作后归位"）不会被关键词截断

### IO3 触摸传感器
- 固件端：IO3 按下 → 14 种随机提示词轮换发送到 LLM
- 服务端：LLM 根据「被摸头反应」提示词每次生成不同动作

### TTS 关闭
- EdgeTTS 连不上外网 (`speech.platform.bing.com`)
- LinkeraiTTS 502 Bad Gateway
- 最终方案：`_TTS_ENABLED = False` 全局关闭，不影响 LLM 推理
